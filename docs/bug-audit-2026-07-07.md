# Exhaustive code audit — 7 Jul 2026

Full-codebase pass requested after the June comparison kept turning up rate discrepancies. Seven independent review passes covered `lib/sitelink.js`, `lib/reportMap.js`, `lib/buildPayload.js`, `lib/pull.js`, the frontend (`app/portal-v2/page.js` + API routes), plus two dedicated cross-checks of the two bug classes that have already bitten this project (sum-vs-average violations, and filters that silently drop more than they should). `scripts/` (150+ one-off diagnostic files) was excluded — not production code.

## Answers your "off by just a little on historical widgets" question

**Confirmed mechanism: `lib/pull.js`'s permanent month-lock.** Once a month rolls from "current" to "previous," any `(site, report)` pair that already has a stored row is *never re-pulled again* — including after a parser bug gets fixed. The month is frozen at whatever `reportMap.js` logic existed the last time it was refreshed, with no mechanism to revisit it.

This means every parser fix from this project's history left a trail: any month locked *before* a fix landed still silently serves the old, wrong numbers today. Concretely:
- **Occupancy** — the `.toFixed()` rounding fix landed 2 Jul 2026. Any month locked before then still rounds rates down at `.x5` boundaries. Small, persistent, matches "off by just a few" exactly.
- **Move-ins & Move-outs** (sourced from ManagementSummary, not the `move_ins_outs` report) — a confirmed ~10x overcount bug existed; October's-scale fix timing vs. lock timing determines which historical months still carry it.
- **Rate per ft² by Customer Type** — fixed *today*. Any month locked before today still reads 0 for that one chart (bigger gap, not "a little," but same disease).

This has already been confirmed to happen once, in this exact codebase, for `insurance_roll` and Customer Churn — both needed a manual targeted re-pull after their parser fix landed, because the lock doesn't self-heal.

**There is no code fix for this** — it's a design tradeoff (closed months shouldn't cost a live SiteLink call every day) that has a real cost (fixes don't retroactively apply). The only remedy is a targeted re-pull of the affected `(report, month)` combinations via the existing `scripts/repull-report-month.js`. Two ways to proceed, your call:
1. **Targeted** — re-pull just `occupancy` and `management` for whichever specific historical months you've actually compared against legacy and found "off." Fast (a few minutes per month).
2. **Full historical re-pull** — re-run `occupancy` + `management` (+ `rent_roll` if you want Rate accuracy fully current too) across all ~122 stored months. Correct everywhere, but this is an hours-long job (same class as the original backfill).

## Fixed already (this session, low-risk/unambiguous — verified with a syntax build after each edit)

1. **`app/portal-v2/page.js` — "Insurance Roll" and "Enquiries by Channel" mock/fallback branches had `live: true`.** Both were almost certainly copy-pasted from the real live branch directly above. This made the app show the green "LIVE" badge on fabricated placeholder numbers (£5,145 / 10.1% / 75.3%, and 94/61/1,210/1,365) whenever the live API was unavailable — the one visual cue that distinguishes mock from real data was missing on exactly the cards where it mattered most. Removed `live: true` from both fallback branches.
2. **`app/portal-v2/page.js` — "Scheduled Reservations vs Scheduled Move-outs" fallback was a frozen real snapshot, not a placeholder.** Every other mock fallback on this page is an obviously-synthetic, `f`-scaled number; this one was the literal 438/267/+171 read pulled live on 7 Jul, unscaled — meaning it looked exactly like a real number and would have silently gone stale as time passed. It's reachable in production twice: the loading skeleton runs on a fixed 700ms timer disconnected from the actual fetch, and any fetch error leaves `liveSites` null indefinitely. Rescaled it by the store filter `f` to match every sibling card's convention.
3. **`app/portal-v2/page.js` — Autobill Conversion showed two different roundings of the same number.** Dashboard/KPIs displayed `.toFixed(1)` (e.g. "86.4%"), Ancillaries displayed `.toFixed(0)` on the identical source field (e.g. "86%"). Standardized Ancillaries to `.toFixed(1)`.
4. **`app/api/bootstrap.js/route.js` — duplicate unit-type rows were silently dropped, not merged.** If two raw SiteLink unit-type rows normalized to the same display name (e.g. two different rows both matching `/office/i`), `if (out[name]) continue` discarded the second row's entire area/rent instead of summing it — unlike every other duplicate-row handler in this codebase, which sums collisions. Fixed to sum.

9. **`buildPayloadRange()` can silently average different sites over different month-counts.** The site-level filter uses `.some()` (survives if *any* month in the range has data) but the inner per-month filter then drops individual gap months before merging — so for a range spanning June, some sites get averaged over 6 months and others over 5, with nothing flagging the mismatch.
10. **June's site gap will spread into July's trend widgets.** Month-over-month deltas default missing baselines to `0` rather than "no data," so L001-L013's July delta tiles will show flat "no change" instead of reflecting a missing prior-month baseline.

## Medium priority

11. **Two KPI-catalog fields are known-unreliable but still user-selectable with no warning:** `insuranceActivity.newPolicies`/`newPremium` (built from the `sNewPolicy` flag, already documented elsewhere as "confirmed £0.00 output even with nonzero move-ins") and `reservations` from InquiryTracking (commented "confirmed unreliable, no longer used" right next to where it's still assigned). Both live only in the custom widget builder's field catalog, not on a default-visible card, so the blast radius is limited to anyone who manually adds these columns — but there's no in-UI caveat if they do. Worth either removing them from the catalog or labeling them.
12. **Month-end date boundary may drop the last day of every month, every report.** The end-date sent to SiteLink is midnight of the last day (start-of-day), not end-of-day. If SiteLink's SOAP API treats the end param as an exact cutoff rather than whole-day-inclusive, every monthly pull is silently missing that day's activity — systematically, forever. Can't be confirmed by reading code; would need a live day-level check against SiteLink.
13. **`CustomReportByReportID`'s "no data" shortcut (`Ret_Code === -1`) is unconfirmed for that report ID specifically** and could be masking a real error as an empty result for True Revenue.
14. **`management` parser: delinquent 30+ totals default to a real `0` (not null) when the source table is missing**, which defeats the intended fallback-to-PastDueBalances chain (`??` can't distinguish "genuinely zero" from "table absent"). Narrow blast radius — one site-month at a time.
15. **`rate_changes.avg_increase_pct` averages each tenant's own already-computed % increase, unweighted** — the one ratio in the codebase that doesn't follow the sum-then-divide convention. May be intentional ("typical tenant" stat) rather than a bug worth fixing; flagging for your call.

## Low priority / cosmetic

17. Merchandise margin can be off by a penny or two after a date-range merge, from summing three independently-rounded monthly figures instead of rounding once at the end.
